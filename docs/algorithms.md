## 1. Публичен ред на листинга на фестивали

**Засяга:** уеб каталог (публичен listing), търсене, карта.

Редът е изцяло управляван от базата данни — не се извършва допълнително скориране в приложния слой.

| Приоритет | Критерий | Посока |
|-----------|----------|--------|
| 1 | `promotion_status = 'promoted'` | promoted > normal |
| 2 | `promotion_rank` | по-голямо = по-напред |
| 3 | Организатор с `plan = 'vip'` | vip > free |
| 4 | `organizer_rank` | по-голямо = по-напред |
| 5 | `start_date` | по-ранна дата напред |

**Кредити за промоция:** един организатор консумира кредит (от `organizer_promotion_credits`) само при преход `normal → promoted`. VIP статусът се проверява спрямо `plan_started_at` / `plan_expires_at`.

## 2. Скорер за персонализирани препоръки (мобилен For You feed)

**Файл:** `lib/recommendations/scorer.ts`  
**Функция:** `scoreFestivalForUser(festival, inputs)`

### Вход

```ts
type RecommendationInputs = {
  followedOrganizerIds: Set<string>;  // UUID-та на следвани организатори
  followedCities:       Set<string>;  // city_slug-ове на следвани градове
  followedCategories:   Set<string>;  // категорийни slug-ове
  recentlyViewedFestivalIds: Set<string>;
  trendingMax: number;    // max saves_count в текущия пул (за нормализация)
  nowIsoDate: string;     // YYYY-MM-DD
};
```

### Формула

```
score = 0

// Персонализация
+ 2.50   ако организаторът е следван
+ 1.40   ако градът е следван
+ 1.15   ако категорията е следвана

// Trending (нормализирано 0–1)
+ trendingNorm × 1.20     (trendingNorm = saves_count / trendingMax, max 1)

// Качество/монетизация
+ 0.45   ако promotion_status = 'promoted'
+ 0.25   ако is_verified = true

// Recency boost (спрямо start_date)
+ 0.22   0–3 дни до старт
+ 0.12   4–10 дни до старт
+ 0.04   11–30 дни до старт
+ 0.00   > 30 дни до старт
- 0.08   вече е започнал (start_date < днес)

// История на потребителя
+ 0.35   ако фестивалът е наскоро разглеждан
```

### Причини (reasons)

Резултатът носи масив `reasons[]` — използва се за обяснителен текст в UI:

| Код | Кога се добавя |
|-----|---------------|
| `because_followed_organizer` | организаторът е следван |
| `popular_near_you` | градът е следван |
| `similar_to_saved` | категорията е следвана |
| `trending_in_city` | `trendingNorm ≥ 0.5` И градът е следван |

### Обяснителни етикети (`lib/recommendations/explanations.ts`)

| Код | Български текст |
|-----|----------------|
| `because_follow` | "Защото следвате {name}" |
| `trending_near_you` | "Трендиращо близо до вас" |
| `starts_soon` | "Започва скоро" |
| `popular_in_city` | "Популярно в {city}" |
| `promoted_pick` | "Подбрано събитие" |
| `recently_updated` | "Наскоро обновено" |

## 3. Онбординг — предложения за категории

**Файл:** `lib/recommendations/onboardingSuggestions.ts`  
**Лимит:** 12 категории

### Скориране

За всеки предстоящ фестивал (до 300, `end_date ≥ днес`):

```
categoryScore[rawCategory] += boost + saves_count

  boost = 3   ако категорията съвпада с избраните/следвани от потребителя
  boost = 1   в противен случай
```

Категориите се сортират по `score DESC`, след това `name ASC` (Bulgarian locale).

## 4. Онбординг — предложения за градове

**Файл:** `lib/recommendations/onboardingSuggestions.ts`  
**Лимит:** 12 града

### Скориране

```
cityScore[slug] = upcomingFestivalCount × 10 + trendingPoints
```

- `upcomingFestivalCount` — брой предстоящи фестивали в града
- `trendingPoints` — сума от `saves_count` на всички фестивали в града

**Сортиране:** `score DESC` → `upcomingCount DESC` → `trendingPoints DESC` → `slug ASC`

## 5. Онбординг — предложения за организатори

**Файл:** `lib/recommendations/onboardingSuggestions.ts`  
**Лимит:** 12 организатора  
**Константи:** `VERIFIED_ONLY_THRESHOLD = 30`

### Прагов механизъм (verified-only threshold)

```
verifiedCount = брой потвърдени организатори в пула

if verifiedCount >= 30:
    organizerRows = само потвърдените
else:
    organizerRows = всички активни (потвърдени + непотвърдени)
```

**Защо:** В ранния период на платформата потвърдените организатори са малко. Ако показваме само тях, качеството на предложенията е лошо (напр. "Читалище с 1 последовател"). При смесен режим потвърдените пак се появяват по-напред заради бонуса от +20 т.

### Формула за скориране

```
score = 0
+ categoryOverlap × 10    (брой съвпадения категории организатор ↔ предпочитания)
+ cityOverlap     ×  8    (брой съвпадения градове организатор ↔ предпочитания)
+ upcomingFestivals ×  6  (брой предстоящи фестивали на организатора)
+ min(40, trendingPoints)  (сума saves_count на фестивалите им, cap 40)
+ min(50, √followers × 8)  (скалиране с корен квадратен: 1→8, 10→25, 100→50)
+ 12   ако има logo_url   (сигнал за profesionalism)
+ 20   ако verified = true
+ 15   ако потребителят вече следва организатора
```

**Защо √followers вместо линейно:** при линеен cap (напр. 30 т.) организатор с 3 последователи и организатор с 3000 са почти неразличими. Корен квадратен дава по-добро разпределение при ниски стойности, без да позволи доминиране при много последователи.

### Обяснителен текст за UI

| Условие | Текст |
|---------|-------|
| Градът съвпада + организаторът има град | "Популярен в {град}" |
| Категорията съвпада (фолклор) | "Организира фолклорни фестивали" |
| Категорията съвпада | "Организира {категория} фестивали" |
| Има предстоящи фестивали | "Има предстоящи събития" |
| Иначе | "Препоръчан организатор" |

### Сортиране

`score DESC` → `upcoming_festival_count DESC` → `followers_count DESC` → `slug ASC`

## 6. Известия — scheduling и dedupe

**Файл:** `lib/notifications/scheduler.ts` + `lib/notifications/dedupe.ts`

### Reminder времена

За всеки запазен фестивал се изчисляват до 2 напомняния:

| Subkind | Базово време | Снапване |
|---------|-------------|---------|
| `24h` | `start_time - 24h` | 08:00–22:00 Sofia |
| `2h`  | `start_time - 2h`  | 08:00–22:00 Sofia |

**Правило за снапване (`normalizeToDayHours`):**
- Ако `hour < 08:00` Sofia → снапва на `09:00` същия ден
- Ако `hour ≥ 22:00` Sofia → снапва на `09:00` следващия ден
- Ако `08:00 ≤ hour < 22:00` → без промяна

Напомняния, чието изчислено `scheduled_for ≥ start_time` или вече е в миналото, се пропускат.

### Dedupe прозорци

| Тип job | Прозорец |
|---------|---------|
| `update` | 60 мин |
| `weekend` | 6 ч |
| `followed_organizer` | 12 ч |
| `new_city` | 24 ч |
| `trending` | 48 ч |
| `reminder` | без прозорец (dedupe само по `dedupe_key`) |

Ако вече съществува `pending` или `sent` job за същия потребител + тип + фестивал/scope в прозореца, новият се пропуска.

### Приоритети

| Тип job | Приоритет |
|---------|----------|
| `reminder` | `high` (или `high` ако старт ≤ 2 дни) |
| `update` | `high` |
| всичко друго | `normal` |

## 7. Известия — rate limiting

**Файл:** `lib/notifications/rateLimit.ts`

### Правила (прозорец 24 часа)

```
ако jobType === 'reminder':
    → никога не се пропуска (reminders са exempt)

ако totalSent >= 2:
    → пропуска се (общ cap 2 нотификации / 24h)

ако jobType IN ('weekend', 'new_city', 'trending') AND promoSent >= 1:
    → пропуска се (промо cap 1 / 24h)
```

**Категории:**
- `totalSent` — всички изпратени (status=`sent`) нотификации за потребителя в последните 24 ч
- `promoSent` — само `weekend`, `new_city`, `trending` job types

Rate limiting се прилага при `insertNotificationJobs()` преди запис в `notification_jobs`.
