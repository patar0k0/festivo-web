Development rules:

• Prefer incremental changes
• Do not invent database schema
• If schema changes are needed, generate a SQL migration in:

scripts/sql/

Migration naming format:

YYYYMMDD_description.sql

Example:

20260305_add_notification_preferences.sql

• Respect Supabase RLS policies
• Do not expose service role keys to the client
• Keep API routes compatible with the existing system
• Avoid breaking changes

Documentation rules:

If a change affects:

• database schema
• system architecture
• notification pipelines
• background jobs
• core modules

Update the corresponding documentation files:

PROJECT_CONTEXT.md

• Never drop tables automatically
• Never drop columns automatically
