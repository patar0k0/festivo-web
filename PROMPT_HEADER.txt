Database schema authority:
• Supabase (live Postgres) is the single source of truth for tables, columns, RPCs, and RLS.
• Follow existing Supabase client usage, queries, and TypeScript types in this repo.
• Do not treat docs/database-schema.md as authoritative; it may be outdated and is not required reading.
• SQL migrations belong in scripts/sql/ (see below); do not invent schema from static docs.

Development rules:

• Prefer incremental changes
• Do not invent database schema
• If schema changes are needed, generate a SQL migration in:

scripts/sql/

Migration naming format:

YYYYMMDD_description.sql

Example:

20260305_add_notification_preferences.sql

• Respect Supabase RLS policies
• Do not expose service role keys to the client
• Keep API routes compatible with the existing system
• Avoid breaking changes

Documentation rules:

If a change affects:

• database schema
• system architecture
• notification pipelines
• background jobs
• core modules

Update the corresponding documentation files:

PROJECT_CONTEXT.md
docs/system-architecture.md
docs/notification-system.md
docs/er-diagram.md

(Schema detail: ship migrations under scripts/sql/ and update architecture/ER docs as needed;
docs/database-schema.md is optional and not part of the mandatory doc set.)

Safety rules:

• Never drop tables automatically
• Never drop columns automatically
• Never delete data automatically
